Quick Start

  1. Open the simulator and select Load Example.
  2. Change the configuration JSON if you want to test a different scenario.
  3. Select strategy choices from the dropdowns, or leave the defaults.
  4. Select Run Simulation.
  5. Review the summary, then download the full report or raw output if needed.

The browser app runs entirely inside the browser. The downloaded raw output is the detailed event log; the downloaded report is the easier file to read first.

Configuration

A configuration is a JSON document with three main sections: simulation, demand, and garage. JSON is strict: keep quotation marks, commas, braces, and square brackets exactly balanced.

Section Meaning Common Changes
simulation How long to run, when the scenario starts, how random results are seeded, and how billing works. Run length, seed, revenue policy, diagnostics.
demand How many vehicles arrive, when demand peaks, and how long vehicles stay. Daily traffic, peak behavior, parking duration.
garage The physical tower layout, elevator, VMRs, preparation positions, and strategy selections. Deck count, tower size, speeds, strategy choices.

Simulation Fields

Field Plain Meaning How To Use It
sessionName Name for this run. Use a short scenario name, such as 3x3-long-stay.
startTime Calendar date and time when simulated time begins. Use ISO-style text such as 2026-06-01T00:00:00-07:00.
durationSeconds Total simulated run length. One day is 86400; thirty days is 2592000.
tickSeconds How much simulated time each update step represents. Keep this at 1 for the current physical model.
timezone Timezone used for days, weekends, and peak windows. Example: America/Los_Angeles.
seed Controls repeatable randomness. Same config plus same seed gives the same generated demand pattern.
outputDir, rawOutputFile Where Node runs write raw output. In the browser, downloads are handled by the browser instead.
revenuePolicy How parking revenue is calculated. chargePerBillingBlock is charged every billingBlockMinutes.
balkingPolicy How likely inbound drivers are to leave when the inbound queue is long. Higher queue thresholds make drivers more patient.
diagnostics Optional technical logging for investigating performance. Usually off for normal users. Turn on when comparing planning performance.

Demand Fields

Field Plain Meaning How It Affects Results
averageInboundPerDay Expected number of inbound vehicles per weekday. Higher values create more arrivals, longer queues, and more work for the garage.
weekendMultiplier Multiplier applied on Saturdays and Sundays. 0.5 means weekend inbound demand is about half of weekday demand.
peakHour Center of the inbound peak period. 9 means the morning peak is centered around 9 AM.
peakWindowHours Width of the peak period. 2 means the peak spans about two hours.
peakShare Fraction of daily inbound demand assigned to the peak window. 0.5 means about half of inbound vehicles arrive during the peak window.
parkingDuration How long vehicles stay before requesting outbound retrieval. minHours, maxHours, and modeHours define a triangular distribution.

Garage Fields

Layout

Field Meaning
rows, columns, floors Size of the parking grid and number of tower floors.
elevatorCell Cell number occupied by the elevator shaft on every floor.
unavailableCells Cell numbers that cannot be used for parking.
streetFacing Whether the tower's long side or short side faces the street. This is recorded but not fully modeled yet.

Elevator, VMRs, and Preparation Positions

Field Meaning
elevator.deckCount Number of elevator decks. Each deck has one VMR.
verticalSpeedMetersPerSecond Elevator vertical movement speed.
floorHeightMeters Vertical distance between floors.
deckRotationSeconds Time for a deck to rotate between garage-facing and street-facing orientation.
vmr.speedMetersPerSecond VMR movement speed inside the garage grid.
vmr.gripReleaseSeconds Time to grip or release a vehicle.
preparationPositions.inboundCount Number of inbound preparation positions where arriving drivers leave vehicles.
preparationPositions.outboundCount Number of outbound preparation positions where drivers pick up vehicles.
kind parallel or sequential preparation-position clearing behavior.
mode designated keeps inbound/outbound assignments fixed. dynamic is reserved for future behavior.
doorSeconds Time for preparation-position doors to open or close.

Strategy Fields

Strategies select the implementation used by the garage. They let scenarios compare different garage behavior while using the same demand pattern.

Strategy Available Value Meaning
placement lowest-access-cost Chooses an empty cell estimated to be easiest to access.
placement first-available Chooses the first available cell in layout order.
retrieval simple-retrieval Baseline retrieval classification and cost estimate.
tripPlanner baseline-physical Builds timed physical trips with deck assignments, blocker moves, VMR paths, elevator stops, and PP transfers.
preparationPositions fixed-assignment Uses the configured inbound/outbound PP assignments.
unblocking idle-after-10-minutes Attempts blocker relocation after ten quiet minutes without inbound or outbound demand.
unblocking disabled Turns off idle unblocking.

How To Interpret The Report

The report has daily rows plus a summary. For short simulations, the report may include a partial final day. The daily array is best for day-by-day behavior; thirtyDaySummary is best for a quick comparison between scenarios.

Report Field Meaning What To Watch
successfulActivities Completed inbound parking plus completed outbound retrieval. Higher is better when demand is comparable.
vehiclesStayingUntilMidnight Vehicles still parked at the end of that calendar day. Useful for understanding carry-over demand.
averageInboundDriverWaitingSeconds Time from generated inbound arrival until the vehicle reaches an inbound PP. This is the driver-facing inbound wait.
averageInboundWaitSeconds Time from generated inbound arrival until the vehicle is placed into a parking cell. This includes internal garage handling after the driver leaves.
averageOutboundWaitSeconds Time from outbound request until the vehicle reaches an outbound PP. This is the driver-facing outbound retrieval wait.
longest...WaitSeconds Worst wait observed that day. Use with averages; a good average can still hide a painful maximum.
biggestInboundQueueLength, biggestOutboundQueueLength Largest queues seen during the day. Queue spikes often explain long waits or balking.
inboundBalkingVehicles Inbound drivers who gave up because the inbound queue was too long. A high value means the system is not keeping up with inbound demand.
balkingOverSuccessfulInboundPercent Balking count compared with successful inbound completions. Can exceed 100% when more drivers balk than successfully park.
maximumOccupancyPercent Highest parking-cell occupancy for the day. Near 100% means capacity pressure and more blocked retrievals are likely.
elevatorTripsCarryingInducedInboundVehicles Trips that had to carry blocker vehicles as induced inbound work. Higher values indicate more blocked retrieval complexity.
totalInducedInboundVehicles Total blocker vehicles moved as induced inbound work. Useful for comparing placement strategies.
idleUnblockingActions, idleUnblockedVehicles Idle maintenance moves performed after quiet periods. Can reduce future retrieval pain, but also consumes mechanical work.
downwardTripPlacements Inbound placements performed during downward trip phases. Shows how often inbound work could not be completed on the upward phase.
totalParkingHours Actual parked time in hours. Operational usage, not necessarily billable time.
totalCollectableParkingHours Billable time after rounding to billing blocks. Used with revenue policy to calculate revenue.
totalElevatorFloorsPassed Total floor-to-floor elevator travel. Proxy for elevator workload and energy use.
totalVmrDistanceMeters Total distance moved by VMRs. Proxy for VMR workload and maintenance.
totalRevenue Revenue calculated from billable parking blocks. Compare only when demand and billing policy are comparable.

Scenario Tips

  • Change one major assumption at a time when comparing two runs.
  • Keep the same seed when comparing strategy changes; this keeps demand comparable.
  • Use different downloaded filenames for scenarios you want to keep.
  • Read averages and longest waits together. Both matter for user experience.
  • High occupancy, high induced vehicles, and high outbound wait usually point to blocked-retrieval pressure.
  • High inbound balking usually means arrival demand is too high for the modeled garage capacity and process speed.
